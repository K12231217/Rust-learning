# blink:分层架构设计思路

> 视角:具有量产经验的嵌入式架构师
> 对象:当前 `blink`(STM32F411 + Embassy)项目
> 姊妹篇:`blink_architecture_learning_from_wg110.md`(向量产固件学什么)、`blink_oop_platform_decoupling_c_to_rust.md`(C 的面向对象思想怎么迁到 Rust)

---

## 0. 这份文档解决什么问题

前两篇分别回答了「向 wg110 量产固件学什么」和「C 的 OOP 思想怎么搬到 Rust」。本文回答最具体的一个:

> **blink 现在这套五文件结构,每个文件为什么存在、边界划在哪、新增功能时往哪放?**

一句话结论:

> **文件边界 = 改它的人的边界。** 不是按「代码类型」分文件,而是按「谁会改它、跟谁一起改」分文件。换板子的人只动 `board.rs`,调策略的人只动 `config.rs`,写业务逻辑的人只动 `app.rs`,它们互不踩脚——这才是分层的价值,不是「文件变多」。

---

## 1. 当前结构总览

```text
src/
  main.rs      系统装配入口:只做初始化、日志、启动任务
  board.rs     唯一硬件事实来源:时钟树、引脚、外设句柄
  config.rs    固件身份 + 可调行为策略
  app.rs       与硬件无关的纯逻辑状态机(可 host 测试)
  runtime.rs   任务间通信通道(Signal 发布系统状态)
  tasks.rs     Embassy async task:只做异步等待 + 驱动硬件
```

六个文件,但只有五种职责。下面逐个说清楚「为什么是它」。

---

## 2. 一张图看清数据流

```text
        ┌─────────────┐
        │   main.rs   │  装配:初始化 → 打日志 → spawn 任务 → 发初始状态
        └──────┬──────┘
               │ Board::init()
        ┌──────▼──────┐
        │  board.rs   │  硬件事实:HSE/PLL/PC13 → 返回就绪的 LedPin
        └──────┬──────┘
               │ 把 led 句柄交给任务
        ┌──────▼──────┐      wait_state()      ┌─────────────┐
        │  tasks.rs   │◄──────────────────────│ runtime.rs  │
        │ blink_task  │                        │  Signal     │
        └──────┬──────┘                        └──────▲──────┘
               │ 问「该闪多快?」                       │ publish_state()
        ┌──────▼──────┐                              │
        │   app.rs    │                       任何任务/main 都能发布
        │BlinkController│ ◄── 读周期常量 ──┐
        └─────────────┘                  │
                                  ┌───────┴─────┐
                                  │  config.rs  │  LED_*_MS / HEARTBEAT_MS
                                  └─────────────┘
```

关键:**箭头都是单向的**。`tasks` 问 `app` 要决策、向 `runtime` 要事件、用 `board` 给的句柄操作硬件,但 `app` / `config` 反过来不认识 `tasks`,更不认识硬件。这是「依赖只能从上往下指」的分层原则。

## 3. 逐层职责与设计理由

### 3.1 `board.rs` —— 唯一硬件事实来源

**职责**:回答「这块板子是什么」。晶振频率、PLL 分频、引脚分配、芯片/板型名,以及把它们初始化成类型安全的句柄(`Board { led }`)。

**为什么单独成层**:硬件改版是必然事件——LED 从 PC13 挪到 PB5、晶振 25M 换 8M、UART 换一组脚。如果这些事实散落在业务代码里,改版就是全工程搜索替换的噩梦。收敛到一处后,改版 = 只动 `board.rs`,`main`/`tasks`/`app` 一行不改。

**一条铁律**:
> 除 `board.rs`(及未来 `drivers/`)外,任何模块都不准知道 LED 接在 PC13、HSE 是 25MHz。

**为什么 `HSE_HZ` 放这里而不是 `config.rs`**:改晶振频率,本来就要同步改它正下方的 PLL 分频(`PllPreDiv` 等)。频率和分频是「一次逻辑改动」,放同一文件、同一屏,才不会改漏。把频率拆到 `config.rs` 反而人为制造了「一次改动、两处文件」的陷阱。

### 3.2 `config.rs` —— 固件身份 + 可调行为

**职责**:回答「这个固件想怎么表现」。版本号、各状态闪烁周期、心跳间隔。

**为什么和 board 分开**:改它的是不同的人、不同的动机。`board.rs` 是硬件负责人在板子改版时改;`config.rs` 是产品/固件负责人在调参数时改(PM 说「心跳灯太快了」、测试想把超时调长)。两类改动频率、风险、责任人都不同,分开后互不干扰。

**判断常量该放哪的标准**:
> 被多个模块共享,或有人(产线/测试/三个月后的你)需要「集中找到它」→ `config.rs`;只有 `board` 一个模块用、且属于硬件事实 → `board.rs`。

**会不会膨胀**:不会因为外设变多而膨胀——外设句柄在 `Board` 结构体、外设参数在 `board.rs`。`config.rs` 只随「可调策略」线性、温和增长。真到 200 行再拆 `config/` 目录(`pub use` 重导出,调用方 `config::XXX` 写法不变),那是膨胀后的解药,不是现在的预防针。

### 3.3 `app.rs` —— 与硬件无关的纯逻辑

**职责**:状态机和策略决策。`SystemState`(Booting/Running/Error)+ `BlinkController`(状态 → 半周期映射)。

**铁律**:本文件**不依赖** `embassy_stm32` / `embassy_time` / 具体 GPIO 类型。周期用 `u64` 毫秒表达,而不是 `embassy_time::Duration`。

**为什么这么做**:这是从 wg110 的 `wg-core` 学到的「可测试 core」思想——
> 能脱离硬件测试的逻辑,就不要绑死在 Embassy task 或 STM32 外设上。

`BlinkController` 不知道 LED 接在哪、不知道用 Ticker 还是 Timer,所以它能在 PC 上跑单元测试(文件末尾已有 `#[cfg(test)]`)。真实产品里业务逻辑会膨胀(读传感器→判断→重试→上报),越多逻辑能在 PC 上测,开发效率和稳定性越高。

### 3.4 `runtime.rs` —— 任务间通信通道

**职责**:用一个全局 `Signal<SystemState>` 解耦任务。谁想改变 LED 行为就 `publish_state(...)`,LED 任务 `wait_state()` 自行响应。

**为什么需要它**:避免任务间网状耦合。反面教材是「按键任务直接操作 LED」——加一个外设就多一条直连,最后变成一团乱麻。正确做法:
> 任务之间不要互相直接调用,而通过清晰的数据通道解耦。
> `button_task` 发布事件 → `led_task` 接收并改模式,两者互不认识。

**为什么用 `Signal` 而不是 `Channel`**:系统状态是「最新值即真相」语义,只关心最后一次状态,不需要排队历史事件——正是 `Signal` 的设计场景。如果将来需要「不能丢、要排队」的事件(如按键序列),再引入 `Channel`。

### 3.5 `tasks.rs` —— 只做异步调度 + 硬件动作

**职责**:Embassy async task。只负责:异步等待、收发事件、驱动硬件、打日志。**不**负责:决定「该闪多快」(那是 `app` 的事)。

**设计要点**:`blink_task` 用 `select` 同时等两件事——ticker 到点(翻转 LED)和 `runtime` 来的新状态(切周期)。决策全部委托给 `BlinkController`,task 自己只是「等→问→做」的执行器。

**为什么用 `Ticker` 而非 `Timer::after`**:Ticker 按绝对时刻对齐,自动补偿循环体耗时,**零累积漂移**。状态变化时重建 Ticker 立即换上新周期。

### 3.6 `main.rs` —— 系统装配器

**职责**:像装配线,不是大杂烩。三个阶段清清楚楚:

```text
Boot:  打身份日志 → Board::init() → 打就绪日志
Start: spawn 各任务(LED 初始进 Booting 快闪)
Run:   延时后 publish_state(Running),切到心跳 —— 演示任务间解耦通信
```

**为什么阶段化**:量产现场排查,启动日志极其重要。要能从 `[boot]/[task]` 日志判断设备卡在哪一阶段。`main` 里不塞业务逻辑,业务在 `app`,调度在 `tasks`。

## 4. 这套结构如何回应「架构是否变好」的检验

参照姊妹篇第 8 节的检验清单:

| 检验问题 | 当前结构的答案 |
|---|---|
| 换 LED 引脚,只改一处? | ✅ 只改 `board.rs` |
| 改闪烁策略,不碰 GPIO 初始化? | ✅ 改 `config.rs` 周期 或 `app.rs` 映射 |
| 加按键任务,不破坏 LED 任务? | ✅ 按键任务 `publish_state`,LED 任务不用动 |
| 加错误状态,不重写主循环? | ✅ `SystemState::Error` 已预留,`publish` 即生效 |
| 业务状态机能脱离硬件测试? | ✅ `app.rs` 无硬件依赖,带 host 单测 |
| 看启动日志知道卡在哪? | ✅ `[boot]/[task]` 分阶段日志 |

## 5. 新增功能时,代码该往哪放(决策速查)

| 你要做的事 | 改哪个文件 | 不该碰 |
|---|---|---|
| 换板子/改晶振/换引脚 | `board.rs` | 其余全部 |
| 调闪烁周期/心跳间隔/版本号 | `config.rs` | — |
| 加一种系统状态、改状态→行为映射 | `app.rs`(+`config.rs` 加周期常量) | `tasks.rs` `board.rs` |
| 加新外设句柄(串口/按键/ADC) | `board.rs` 的 `Board` 结构体 | — |
| 加新的 async 任务 | `tasks.rs` + `main.rs` spawn | `app.rs` |
| 让任务间传递新事件 | `runtime.rs` 加 Signal/Channel | 任务互相直连 |
| 接入错误处理路径 | 检测处 `publish_state(Error)` | 重写主循环 |

**一个反模式提醒**:不要因为「方便」就在 `tasks.rs` 里直接写死周期、或在 `main.rs` 里 `Output::new(p.PC13, ...)`。这会把硬件事实和业务策略重新打散,正是这套分层要消除的东西。

## 6. 演进路线:现在不做什么

这套结构是「单 crate 内的职责分层」,刻意保持轻量。以下是**故意没做**的,以及触发条件:

| 暂不引入 | 触发条件 |
|---|---|
| `drivers/` 目录 | `board.rs` 因外设驱动变多而臃肿时,把具体器件驱动拆出去 |
| `config/` 目录 | `config.rs` 超 ~200 行或多人 merge 冲突时 |
| 编译期多板型(feature flag) | 真出现第二块板型时 |
| workspace 拆分 | 出现多 app、可复用驱动、Bootloader、需要独立 host test crate 时 |
| Bootloader / OTA | 最后再做,且要先设计 Flash layout 和升级协议 |

> workspace 是复杂度的结果,不是架构高级的原因。一个 LED 项目强行 workspace 只会增加维护成本。

---

## 7. 一句话总结

> blink 的分层不是「把代码分很多文件」,而是让**硬件事实、可调策略、纯逻辑、任务调度、任务通信、系统装配**各有清晰边界。每条边界都对应「谁会改它、何时改、跟谁一起改」。当新增功能时你能毫不犹豫地知道代码该放哪——这套架构就成功了。
